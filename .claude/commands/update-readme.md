# Update README

Scan the codebase and update all documentation files to accurately reflect the current state of the project.

## Worktree Setup

Before starting any work, ensure you are operating in a git worktree:

1. **If already in a worktree** (current directory path contains `.claude/worktrees/`): proceed in the current directory.
2. **If NOT in a worktree**: Create one:
   ```bash
   git worktree add .claude/worktrees/update-readme-$(date +%s) -b update-readme-$(date +%s)
   ```
   Then `cd` into the worktree directory before proceeding.

**All file reads, edits, and commits must happen inside the worktree.** Never modify files in the main working directory.

## Scope

Update these documentation files:

1. **Root `README.md`** — Main project documentation
2. **`admin/README.md`** — Admin panel documentation
3. **`docs/contributing.md`** — Code conventions, quality gates, dependency notes

**Skip**: `.expo/README.md` files (auto-generated by Expo), `docs/data-pipeline.md`, `docs/cloudflare-custom-domain-setup.md` (infrastructure-specific, not affected by code changes).

## Phase 1 — Audit Current State

Gather the current state of the codebase by inspecting these areas. Do NOT modify any files during this phase.

### 1.1 Project Structure

Scan the directory tree to verify `README.md` sections match reality:

- `app/` — list all route files and tab screens
- `src/` — list all top-level subdirectories and their purpose
- `admin/src/` — list all top-level subdirectories and their purpose
- `shared/` — list all exported modules
- `supabase/migrations/` — count migration files
- `scripts/` — list all scripts
- `docs/` — list all documentation files

### 1.2 Tech Stack Versions

Check actual versions from `package.json` and `admin/package.json`:

- React Native, Expo SDK, Expo Router
- Zustand, TanStack Query
- Next.js, Tailwind CSS, React Hook Form, TanStack Table
- Jest, Vitest
- TypeScript
- Any other key dependencies listed in the README

### 1.3 Test Coverage

Count current test suites and test cases:

```bash
# Mobile
npx jest --silent --forceExit 2>&1 | tail -5

# Admin
cd admin && npx vitest run 2>&1 | tail -10
```

### 1.4 Database Tables

Read the latest migration files to check if any new tables have been added or existing tables have been renamed/removed since the README was last updated.

### 1.5 Environment Variables

Check `.env.example` and `admin/.env.example` (if they exist) and compare against the README's environment variable tables.

### 1.6 Scripts and Commands

Check `package.json` scripts sections in both root and admin to verify documented commands are accurate.

### 1.7 Features

Scan `app/(tabs)/` for tab screens, `app/` for other screens, and `admin/src/app/(dashboard)/` for admin features to verify the feature lists are accurate.

### 1.8 Contributing Guide Accuracy

For `docs/contributing.md`, verify:

- **300-line rule**: Exemption list matches `CLAUDE.md` (the source of truth)
- **Test conventions**: Framework versions, mock patterns, and flags match actual config
- **ESLint/Prettier config**: Settings match `eslint.config.js`, `.prettierrc`, and `admin/eslint.config.mjs`
- **Path aliases**: Alias table matches `tsconfig.json` and `admin/tsconfig.json`
- **Dependency constraints table**: Version constraints still apply (check actual `package.json` versions)
- **File organization examples**: Directory structure examples match actual codebase layout
- **Quality gate commands**: Match the commands in `CLAUDE.md`

## Phase 3 — Update READMEs

Apply the approved updates. Follow these guidelines:

### Root README.md

- Keep the existing structure and tone
- Update version numbers to match actual `package.json` values
- Update test counts to match actual output
- Add/remove features based on actual screens found
- Update project structure tree to match actual directories and files
- Update database table list if new tables exist
- Update environment variable tables if new vars are needed
- Do not remove sections — only update or add content
- Keep all existing links intact unless they point to non-existent files

### admin/README.md

If it still has the default create-next-app template, replace it entirely with project-specific content covering:

- Project description (Faniverz Admin Panel)
- Tech stack (from `admin/package.json`)
- Prerequisites
- Setup instructions (install, env vars, dev server)
- Project structure (`admin/src/` directory tree)
- Available scripts (`admin/package.json` scripts)
- Testing (`vitest run`, test conventions)
- Key features (list admin dashboard pages from `admin/src/app/(dashboard)/`)
- Quality gates (`npx eslint . --max-warnings 0 && npx tsc --noEmit && npx vitest run`)

### docs/contributing.md

- Keep it in sync with `CLAUDE.md` — the contributing guide is the developer-facing version of the same rules
- Update the 300-line exemption list if `CLAUDE.md` has changed
- Update test convention sections (framework versions, mock patterns, flags)
- Update the dependency constraints table — remove entries for constraints that no longer apply, add new ones
- Update path alias table if `tsconfig.json` has changed
- Update file organization examples if the actual directory structure has diverged
- Update Prettier settings if `.prettierrc` has changed
- Update quality gate commands to match `CLAUDE.md`
- Do not duplicate content from `README.md` — link to it instead where appropriate

### Writing Style

- Use clear, concise technical writing
- Prefer tables for structured data (versions, env vars, ports)
- Use fenced code blocks with language tags for all commands
- Keep the README under 500 lines — link to `docs/` for detailed guides
- Do not add badges, shields, or decorative elements unless they already exist

## Phase 4 — Verify

After updating, verify:

1. All internal links in the documentation files point to files that exist (e.g., `admin/README.md`, `docs/contributing.md`)
2. All documented commands are valid (spot-check 2-3 commands)
3. No broken markdown formatting (tables, code blocks, headers)
4. `docs/contributing.md` does not contradict `CLAUDE.md` — if there is a conflict, `CLAUDE.md` is the source of truth
5. No duplicate content between the three files — each should have its own focus:
   - `README.md`: Project overview, setup, architecture
   - `admin/README.md`: Admin-specific setup, features, API routes
   - `docs/contributing.md`: Code conventions, quality standards, dependency notes

Report any broken links or contradictions found.

## Rules

- Never fabricate version numbers — always read from `package.json`
- Never fabricate test counts — always run the test suites
- Never remove existing sections without user approval
- Never add promotional language, badges, or emojis
- Keep factual and developer-focused
- If a README section references something you cannot verify (e.g., deployment URLs, API keys), leave it unchanged
- Do not modify any source code files — this skill only updates documentation files (README.md, contributing.md)
- `CLAUDE.md` is the source of truth for conventions — `docs/contributing.md` must not contradict it
