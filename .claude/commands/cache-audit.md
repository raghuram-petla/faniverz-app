# Cache Invalidation Audit

Audit all TanStack Query mutations and queries in the admin panel to find cache invalidation gaps — places where a mutation modifies data that affects a query's results, but doesn't invalidate that query's key. Runs in a **find-fix loop until clean**.

## Loop Mode

This skill runs in a **loop until clean**. After completing a full scan-report-fix cycle, immediately start a new scan from Phase 1. Keep looping until **3 consecutive runs find zero gaps**. Track the run counter:

```
Run 1 → found 8 gaps → fix → Run 2 → found 2 gaps → fix → Run 3 → found 0 → Run 4 → found 0 → Run 5 → found 0 → DONE (3 consecutive clean runs)
```

**Rules for loop mode:**

- Each run is a full Phase 1–4 cycle (inventory, cross-reference, report, fix)
- A "clean run" means zero cache invalidation gaps found
- The 3-clean-run counter resets to 0 if any gaps are found
- Between runs, re-read all modified hook files to pick up changes from previous fixes
- Between runs, print: `### Run N complete — {X gaps found | clean} (consecutive clean: M/3)`
- After 3 consecutive clean runs, print: `### Cache Audit complete — 3 consecutive clean runs achieved`

**CRITICAL — No shortcuts allowed:**

- **Every single run MUST perform a full scan.** You MUST actually read the hooks and cross-reference queries vs mutations in every iteration — no skipping, no "the code hasn't changed so it's clean", no assuming previous runs were thorough enough.
- **Launch Explore or general-purpose Agents** for each scan to ensure independent, thorough review. Do not rely on memory of previous runs.
- **Never declare a run "clean" without reading the actual code.** If you cannot prove you read the hooks in that run, it doesn't count.

## Phase 1 — Inventory

### 1a. Inventory all query keys

Scan `admin/src/hooks/` for every `useQuery` / `useInfiniteQuery` call. Record each query key and what data it fetches. Include:

- Direct calls in hook files
- Calls generated by `createCrudHooks` (usePaginatedList, useSimpleList, useSingle)
- Calls generated by `createMovieChildHooks` (useList)

### 1b. Inventory all mutations

Scan `admin/src/hooks/` for every `useMutation` call. Record:

- What table/data each mutation modifies
- Which query keys it invalidates in `onSuccess` / `onSettled`
- Any `extraInvalidateKeys` passed to factory functions

Include mutations from:

- Direct `useMutation` calls
- `createCrudHooks` (useCreate, useUpdate, useDelete)
- `createMovieChildHooks` (useAdd, useUpdate, useRemove)
- `createSimpleMutation` factory
- Composite handlers (`useMovieEditHandlers.ts` handleSubmit/handleDelete, `useMovieAddState.ts` handleSubmit)

## Phase 2 — Cross-Reference for Gaps

For each mutation, determine ALL queries whose results could change because of that mutation. Check if the mutation invalidates those query keys.

**Key rules:**

- TanStack Query uses **prefix matching** by default (no `exact: true`), so `invalidateQueries({ queryKey: ['admin', 'movies'] })` WILL match `['admin', 'movies', search, filter]`
- A mutation that changes a table's row count affects any query that counts rows in that table (e.g., `useDashboardStats`)
- A mutation on a junction table (movie_platforms, movie_production_houses) affects queries that JOIN through that table
- A mutation that changes a field used in another query's WHERE clause or display affects that query

**Key patterns to watch for:**

- Mutations on junction tables (movie_production_houses, movie_platforms) that affect scoped queries
- Child mutations (theatrical runs, posters) that affect parent movie display or cross-page views
- The `['admin', 'platform-movie-ids']` prefetch cache used by useAdminMovies
- Theater-related query keys: `theater-movies`, `upcoming-movies`, `upcoming-rereleases`, `theater-search`
- `setMainPoster` which updates both `movie_images.is_main_poster` AND `movies.poster_url`
- Dashboard stats (`['admin', 'dashboard']`) which counts movies, actors, users, reviews, feed items, comments
- Mutations that modify `movies.in_theaters`, `movies.release_date`, or `movies.status` — these affect theater views

## Phase 3 — Report

For each gap found, report:

```
## Cache Audit Report (Run N)

### Gaps Found

| # | File | Hook/Mutation | Missing Invalidation Key | User-Visible Symptom | Severity |
|---|------|---------------|--------------------------|---------------------|----------|

### Summary
- Critical: X | High: Y | Medium: Z | Total: N
```

**Severity levels:**

- **CRITICAL**: Wrong data actively displayed to user (e.g., stale count, wrong poster)
- **HIGH**: Stale data visible until manual page refresh
- **MEDIUM**: Stale for staleTime duration, auto-refreshes eventually

**Do NOT wait for user confirmation.** Proceed directly to Phase 4 and fix ALL gaps.

## Phase 4 — Fix

Fix all gaps in order of severity (Critical first).

### Fix Workflow

1. **Apply minimal fix** — add missing `invalidateQueries` calls
2. **Prefer `extraInvalidateKeys`** on `createMovieChildHooks` / `createCrudHooks` when applicable (centralizes the invalidation)
3. **Add `@sideeffect` annotations** explaining why the invalidation is needed
4. **Run quality gates** after all fixes:
   - `cd admin && npx tsc --noEmit && npx vitest run`
5. **Check 300-line limit** on all modified files
6. **Do NOT commit** — leave changes uncommitted for the user to review

After fixing, loop back to Phase 1 with a new scan.

## Final Report

After 3 consecutive clean runs, print:

```
## Cache Invalidation Audit — Complete

### Gaps Fixed (across all runs)
| # | Severity | File | Mutation | Missing Key Added | Symptom Prevented |
|---|----------|------|----------|-------------------|-------------------|

### Quality Gates
- Admin: PASS/FAIL (X suites, Y tests)

### Stats
- Total runs: N
- Total gaps found and fixed: M
- Consecutive clean runs: 3/3
```

## Rules

- **Admin panel only** — scope is `admin/src/hooks/`
- **Minimal fixes** — only add missing invalidation calls, don't refactor surrounding code
- **No new features** — only fix cache invalidation gaps
- **Prefer factory options** — use `extraInvalidateKeys` over manual `invalidateQueries` when a factory supports it
- **Don't commit** — leave all changes for user review
- **Code-intel annotations** — add `@sideeffect` annotations to new invalidation calls per CLAUDE.md rules
